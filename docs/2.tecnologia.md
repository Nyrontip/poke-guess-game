# 🛠️ Tecnología — Poke Guess Game

## Stack completo

```
Frontend (Solid.js + Vite)
    │
    ▼ HTTP (REST API)
Backend (PHP 8.2 — vanilla, sin framework)
    │
    ▼ pdo_pgsql
Base de datos (PostgreSQL 15)
```

## Frontend

| Tecnología | Versión | Propósito |
|-----------|---------|-----------|
| [Solid.js](https://www.solidjs.com/) | ^1.9.5 | UI reactiva sin Virtual DOM. Elegido por rendimiento y simplicidad vs React. |
| [Vite](https://vitejs.dev/) | ^7.1.4 | Bundler y dev server con HMR. |
| [nanoid](https://github.com/ai/nanoid) | ^4.0.2 | Generación de IDs únicos para players y scores. |
| Vite proxy | — | Redirige `/api/*` al backend en desarrollo (`localhost:8000`). |

**No se usa:** TypeScript (es JSX plano), sql.js (era del plan original, hoy la DB es PostgreSQL).

### Estructura

```
frontend/src/
  ├── App.jsx          ← Router casero con Show + estado route
  ├── main.jsx         ← Entry point
  ├── lib/
  │   ├── api.js       ← Wrapper de PokeAPI con cache en sessionStorage
  │   └── db.js        ← Cliente REST del backend PHP
  ├── pages/
  │   ├── Home.jsx     ← Selección/registro de jugador + historial
  │   ├── Game.jsx     ← Lógica del juego, timer, opciones
  │   └── Results.jsx  ← Resumen, guardar score, leaderboard
  └── styles.css       ← Todos los estilos en un archivo
```

## Backend

| Tecnología | Versión | Propósito |
|-----------|---------|-----------|
| PHP | 8.2 FPM (Alpine) | API REST, vanilla sin framework |
| PDO | — | Conexión a PostgreSQL con prepared statements |
| php.ini personalizado | — | `memory_limit=256M`, `display_errors=On` (dev), UTC |

### Estructura

```
backend/src/
  ├── index.php               ← Router con regex matching
  ├── Database.php            ← Singleton PDO con lazy connection
  ├── PlayerController.php    ← CRUD de jugadores
  └── ScoreController.php     ← CRUD de scores con filtro por período
```

**Decisiones técnicas:**
- **Sin framework**: el proyecto es pequeño (4 endpoints), un router de 40 líneas alcanza.
- **Singleton en DB**: conexión única, reutilizada en toda la request. Correcto para FPM donde cada request es un proceso fresco.
- **Prepared statements**: todas las queries usan PDO con bind de parámetros. Protección contra SQL injection.
- **`exit` en respuestas**: atajo directo para evitar el overhead de un response object. Simplifica pero hace el código intesteable.

## Infraestructura

| Componente | Imagen | Rol |
|-----------|--------|-----|
| **Nginx** | `nginx:alpine` | Reverse proxy. Sirve SPA estática + deriva `/api/*` a PHP-FPM |
| **PHP-FPM** | `php:8.2-fpm-alpine` | Ejecuta el backend PHP, escucha en puerto 9000 |
| **PostgreSQL** | `postgres:15-alpine` | Base de datos relacional |
| **Docker Compose** | — | Orquestación de los 3 servicios + healthcheck |

### Puertos

| Servicio | Puerto host | Puerto container |
|----------|------------|-----------------|
| Nginx (app) | 8000 | 80 |
| PostgreSQL | 5433 | 5432 |

### Configuración de red

Toda la comunicación interna es por la red de Docker:
- `php:9000` — Nginx → PHP-FPM (FastCGI)
- `postgres:5432` — PHP → PostgreSQL (TCP)

## Entorno de desarrollo

- **Frontend**: `npm run dev` → Vite en `:3000` con proxy a `:8000`
- **Backend**: `docker compose up` → PHP + PostgreSQL + Nginx en `:8000`
- **Hot reload**: solo frontend (Vite HMR). Para PHP se refresca el navegador (FPM recarga en cada request).

## Despliegue

- **Frontend build**: `npm run build` → `dist/` servido por Nginx como SPA
- **Vercel**: config `vercel.json` presente para deploy del frontend standalone (sin backend)
